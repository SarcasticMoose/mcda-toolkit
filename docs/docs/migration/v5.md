# Migrating to v5.x

Version 5.x introduces the **Pipeline** as the primary way to run a decision analysis. This is a breaking change in the intended usage pattern — not necessarily in the individual APIs, but in how the process is structured and executed.

## What changed

In previous versions, the caller was responsible for driving the entire evaluation flow manually:

```csharp
// Before (< 5.x)
var data = new DataBuilder()
    .AddWeights(weights)
    .AddDecisionCriteria(types)
    .AddDecisionMatrix(matrix)
    .Build();

var method = Promethee2Builder
    .Create()
    .WithNormalizationMethod(NormalizationMethod.MinMax)
    .WithPreferenceFunction(PreferenceFunction.Unnamed)
    .Build();

var result = method.Run(data);
```

Each step was independent. The caller connected them. There was no shared structure representing the process itself.

Starting from 5.x, the same flow is expressed through the Pipeline:

```csharp
// After (5.x+)
var result = PipelineBuilder
    .Create()
    .AddNormalization(NormalizationMethod.MinMax)
    .Build()
    .Execute(data, method);
```

The individual building blocks — `DataBuilder`, method builders — remain unchanged. What changed is that they are now passed into a pipeline rather than wired together manually by the caller.

## Migration steps

1. **Keep your existing data and method setup as-is.** `DataBuilder` and all method builders are unchanged. No modifications are needed there.

2. **Replace the direct `method.Run(data)` call** with a `PipelineBuilder` that receives both the data and the method, then call `Execute()` on the built pipeline.

3. **Remove any manual coordination logic** between steps. If you were passing results from one stage into the next yourself, the pipeline now handles that.

## Normalization

Prior to 5.x, normalization methods handled cost and benefit criteria internally. A single normalization class was responsible for both scaling the values and accounting for the direction of each criterion. This created implicit coupling — the normalization formula and the transformation logic were entangled in the same place.

Starting from 5.x, these responsibilities are fully separated:

| | Before (< 5.x) | After (5.x+) |
|---|---|---|
| Normalizer | Scaled values + handled cost/benefit | Scales values only |
| Transformation | Embedded inside normalizer | Handled by a dedicated transformer |
| Criteria awareness | Yes — normalizer inspected criteria types | No — normalizer is direction-agnostic |

The transformer now runs as a separate preprocessing step before normalization. See [Normalization](../usage/normalization) for details on the new architecture.

## Calling `Run()` directly is no longer possible

`IMcdaMethod<T>.Run()` has been removed from the public API. Code that called it directly will not compile against 5.x.

The Pipeline is now the only supported way to execute a decision analysis. There is no workaround or compatibility shim — migration is required.
